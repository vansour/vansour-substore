# Stage 1 Baseline

## 目标

阶段一用于冻结当前 `submora` 的外部行为和重写边界，为后续 `Dioxus 0.7.3 + Axum` 重写提供唯一基线。

本阶段只定义“必须兼容什么”和“明确不兼容什么”，不启动新架构实现。

## 当前产品定义

当前系统是一个单二进制 Rust 服务，负责三类能力：

- 提供基于 Cookie Session 的管理后台。
- 提供管理员账号、用户、链接、排序等管理 API。
- 提供公共纯文本聚合接口 `GET /{username}`。

## 当前数据与运行基线

### 当前数据库模型

当前实现使用 SQLite，至少包含以下两张核心表：

- `admins(username, password_hash, created_at, updated_at)`
- `users(username, links, rank, created_at)`

其中：

- `admins.username` 是当前唯一管理员标识。
- `users.username` 直接作为公共路由 `/{username}` 的路径参数。
- `users.links` 当前以 JSON 数组字符串形式存储。
- `users.rank` 控制后台展示顺序和默认聚合顺序。

### 当前默认运行参数

当前实现默认值如下：

- 监听地址：`0.0.0.0`
- 监听端口：`8080`
- SQLite 最大连接数：`5`
- 远程抓取超时：`10s`
- 默认并发抓取数：`10`
- 每用户默认最大链接数：`100`
- 默认最大用户数：`100`

### 当前默认初始化行为

- 数据库为空时自动创建默认管理员。
- 默认管理员用户名为 `admin`，密码为 `admin`。
- 管理后台默认入口为 `/`。
- 公共订阅入口格式为 `http://host/{username}`。

## 重写范围

重写后仍然保留以下产品定位：

- 同一站点同时提供管理后台和 API。
- 管理后台用于维护用户和每个用户的订阅链接。
- 公共接口继续按用户聚合链接内容并返回纯文本。
- 默认部署模型仍然是单服务对外监听 HTTP 端口。

## 必须保持兼容的外部行为

### 访问入口

- `GET /` 返回管理后台入口页面。
- `GET /healthz` 返回 `200 OK` 和纯文本 `ok`。
- `GET /{username}` 保留为公共订阅聚合入口。

### 认证模型

- 管理接口继续基于 Cookie Session 鉴权。
- 未登录访问管理接口时继续返回 `401 Unauthorized`。
- 登录成功后由服务端设置 Cookie，前端不自行持久化 token。

### 管理 API 前缀

以下前缀必须保留：

- `/api/auth/*`
- `/api/users*`

### 基本管理能力

重写后首版必须继续支持：

- 管理员登录、登出、查询当前登录身份。
- 修改管理员用户名和密码。
- 创建用户、删除用户、查看用户列表。
- 查看和更新某个用户的订阅链接。
- 调整用户排序。
- 从后台复制并打开公共聚合链接。

### 公共聚合接口

`GET /{username}` 必须继续满足：

- 成功时返回 `Content-Type: text/plain; charset=utf-8`。
- 输出内容按用户配置顺序拼接。
- 空链接列表时返回空文本而不是 HTML 页面。
- 用户不存在时返回 `404`。

### 基本运行约束

- 默认监听地址仍然兼容 `0.0.0.0:8080`。
- 默认健康检查路径仍然是 `/healthz`。
- 默认数据存储仍然兼容 SQLite。
- 容器默认对外暴露 `8080`。
- 首版仍采用“后端统一托管前端页面和 API”的单入口模型。

## 允许改变的内部实现

以下内容允许在重写中调整，不属于兼容承诺：

- 前端技术栈从静态 HTML/JS 改为 Dioxus 0.7.3。
- 后端内部模块组织、crate 划分、服务层设计。
- SQLite 表结构，前提是完成数据迁移。
- 日志实现、限流实现、HTML 转文本实现细节。
- 前端视觉设计和组件结构。

## 明确不需要继承的当前缺陷

以下是当前实现中的已知问题，重写时应修复，而不是保持兼容：

- 链接保存去重时可能打乱原有顺序。
- 前端对后端错误处理不严格，可能误报成功。
- 账号更新时当前密码校验不够严格。
- `max_users` 约束未在创建用户路径中完整生效。
- SSRF 防护未覆盖重定向链的再次校验。
- 响应体大小限制过度依赖 `Content-Length`。

## 首版重写非目标

以下内容不作为首版重写交付要求：

- 多管理员和 RBAC。
- 任务队列、抓取缓存、定时刷新。
- 历史版本、审计日志、回滚。
- 多租户、组织空间、开放 API token。
- SSR 页面渲染。

## 切换验收标准

新版本切换前，必须满足：

- 旧链接 `GET /{username}` 不需要变更客户端配置即可继续访问。
- 管理后台可完成当前全部核心操作。
- 管理 API 的路径和基本语义不变。
- 默认 Docker 启动后仍可通过 `http://127.0.0.1:8080` 访问。
- 健康检查仍可通过 `/healthz` 验证。

阶段一完成后，仓库内至少应存在：

- 当前系统外部行为基线。
- 兼容接口清单。
- 不继承缺陷清单。
- 后续阶段的验收边界。
